## 一、产品概述

### 1.1 产品定位

ZigSight 是一款面向智能家居/IoT 行业的 **Zigbee 网络可视化诊断平台**，帮助工程师快速定位网络问题、优化网络部署、提升产品稳定性。

> **一句话描述：** 让 Zigbee 网络问题"看得见、查得到、修得好"。

### 1.2 背景与痛点

| 痛点 | 现状 | 影响 |
|---|---|---|
| 网络问题黑箱 | 设备掉线/延迟无根因，只能盲猜 | 排障时间长，客户投诉多 |
| 拓扑不可见 | 不知道 Router 之间怎么连接的 | 部署不合理，网络脆弱 |
| 干扰无法量化 | 不知道 WiFi 对 Zigbee 影响多大 | 信道选择靠经验，不稳定 |
| 问题不可复现 | 昨天的网络问题今天看不到了 | 无法追溯，重复踩坑 |
| 缺乏专业工具 | Ubiqua/Perytons 只能抓包，不诊断 | 分析靠人工，效率低 |

### 1.3 目标用户

| 角色 | 场景 | 核心诉求 |
|---|---|---|
| **嵌入式工程师** | 固件开发、协议栈调试 | 底层数据透明，能定位到帧级问题 |
| **测试工程师** | 量产测试、稳定性测试 | 自动化诊断，一键出报告 |
| **技术支持/FAE** | 客户现场排障 | 快速定位问题，给出解决方案 |
| **项目经理** | 产品质量把控 | 网络健康度量化，趋势追踪 |

### 1.4 核心价值

```
传统排障流程:                    ZigSight 流程:
客户反馈掉线                     客户反馈掉线
  → 猜测原因                       → 打开 ZigSight
  → 去现场抓包(1-2天)              → 看拓扑+告警(5分钟)
  → 分析日志(半天)                 → 定位根因
  → 尝试修复                       → 针对性修复
  → 不确定修好没                   → 验证修复效果
总计: 2-3天                       总计: 1-2小时
```

## 四、数据规格

### 4.1 采集数据源

| 数据源 | 采集方式 | 频率 | 数据内容 |
|---|---|---|---|
| Coordinator 邻居表 | EZSP `getNeighborTable` | 10s | NWK/IEEE/LQI/深度/关系 |
| Coordinator 路由表 | EZSP `getRoutingTable` | 10s | 目的/下一跳/状态/老化 |
| Coordinator 子节点表 | EZSP `getChildData` | 10s | 子节点 NWK/类型 |
| Router 邻居表 | 自定义 ZCL 0xFC00 上报 | 60s (可配) | 邻居 NWK/IEEE/LQI |
| Router 路由表 | 自定义 ZCL 0xFC00 上报 | 60s (可配) | 路由信息 |
| ED Scan | 自定义 ZCL 命令触发 | 按需 | 信道/RSSI |
| 帧级数据 | 被动嗅探 (可选) | 连续 | MAC/NWK/APS 帧详情 |

### 4.2 存储策略

| 数据类型 | 原始保留 | 聚合保留 | 存储引擎 |
|---|---|---|---|
| 拓扑快照 | 7天 | 1min/30天, 10min/90天, 1h/永久 | SQLite |
| ED Scan | 7天 | 同上 | SQLite |
| 事件日志 | 90天 | - | SQLite |
| 帧级数据 | 24h | - | 文件 (可选) |

### 4.3 数据输出格式

标准 JSON 快照格式（所有模块共享）：

```json
{
  "timestamp": "ISO8601",
  "coordinator": {"ieee": "...", "nwk": "0x0000"},
  "nodes": [{"nwk": "...", "type": "...", "lqi": 0, "status": "online"}],
  "links": [{"source": "...", "target": "...", "lqi": 0}],
  "alerts": [{"type": "...", "severity": "...", "message": "...", "node": "..."}]
}
```

## 五、用户流程

### 5.1 典型使用流程

```
1. 连接 Coordinator
   ┌──────────────┐
   │ 选择串口/波特率 │ → 连接 → 自动开始采集
   └──────────────┘

2. 实时监控
   ┌──────────────┐
   │ 查看拓扑图     │ → 发现红色告警 → 点击查看详情
   │ 查看信道热力图 │ → 发现 WiFi 干扰 → 查看推荐信道
   └──────────────┘

3. 问题诊断
   ┌──────────────┐
   │ 告警面板       │ → 按严重程度排序 → 点击跳转到相关节点
   │ 诊断建议       │ → "建议在 X 位置增加中继 Router"
   └──────────────┘

4. 历史回溯
   ┌──────────────┐
   │ 打开时序回放   │ → 拖到问题发生时间 → 查看拓扑变化
   │ 导出问题报告   │ → 发给研发/客户
   └──────────────┘
```

### 5.2 核心用户旅程

**场景 A：客户反馈"设备经常掉线"**

```
Step 1: 连接 Coordinator → 查看拓扑
Step 2: 发现某 SED 连线颜色为红色 (LQI=35)
Step 3: 点击 SED → 查看详情 → parent 是远处 Router
Step 4: 告警面板提示 "SED 0x5678 LQI 极弱，建议迁移到近处 Router"
Step 5: 在 SED 附近增加常供电 Router → 等待重新入网
Step 6: 刷新拓扑 → LQI 变绿 → 问题解决
```

**场景 B：批量部署后网络不稳定**

```
Step 1: 连接 Coordinator → 切换到信道分析
Step 2: 热力图显示信道16-19 RSSI=-40dBm (严重)
Step 3: 诊断: "疑似 WiFi 6 干扰"
Step 4: 推荐: "信道25最佳 (RSSI=-82dBm)"
Step 5: 迁移信道 → 网络稳定
```

## 六、非功能性需求

### 6.1 性能

| 指标 | 目标 |
|---|---|
| 支持节点数 | ≤ 200 节点 |
| 拓扑刷新延迟 | ≤ 2s |
| 历史数据加载 | 24h 数据 ≤ 3s |
| 内存占用 | ≤ 512MB |
| SQLite 数据库大小 | ≤ 500MB/周 |

### 6.2 兼容性

| 项目 | 支持 |
|---|---|
| 协调器芯片 | EFR32MG21 / MG24 (EmberZNet) |
| 协议栈 | EmberZNet 6.x+ |
| 操作系统 | Windows 10+ / macOS 12+ / Ubuntu 20.04+ |
| 浏览器 | Chrome 90+ / Edge 90+ / Firefox 90+ |
| Python | 3.9+ |

### 6.3 安全

- 串口通信仅限本地，不暴露到公网
- 数据存储本地 SQLite，不上传云端
- 未来企业版可考虑加密存储 + 用户认证

### 6.4 部署

| 方式 | 说明 | 优先级 |
